# Проект SmartHome

## Задание 1. Анализ и планирование

Анализ выполнен на основе:

- описания кейса компании «Тёплый дом»,
- требований к целевой экосистеме,
- предоставленного кода текущего монолитного приложения (Go + PostgreSQL).

Для структурирования материала анализ разделён на следующие документы:

1. [Бизнес-контекст и требования](./docs/architecture/01-business-context.md)
2. [Архитектурные допущения и принципы проектирования](./docs/architecture/02-architecture-foundations.md)
3. [Нефункциональные требования](./docs/architecture/03-non-functional-requirements.md)

Такое разделение позволяет:
- отделить фактическое состояние системы (AS-IS) от проектных решений (TO-BE),
- явно зафиксировать принятые архитектурные гипотезы,
- упростить дальнейшую декомпозицию на микросервисы.

При анализе использовались принципы Domain-Driven Design (DDD):

- выявление доменных областей,
- определение bounded contexts,
- разделение бизнес-логики и инфраструктурных компонентов,
- подготовка к последующей декомпозиции на микросервисы.

### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователь может удалённо включать и выключать отопление в доме.
- Управление осуществляется централизованно через сервер приложения.

**Управление датчиками (сенсорами):**

- Система получает данные о температуре с датчиков, установленных в доме (по факту через внешний Temperature API.).
- Доступны операции:
    - получение списка сенсоров `GET /sensors`,
    - получение сенсора по ID `GET /sensors/:id`,
    - создание сенсора `POST /sensors`,
    - обновление сенсора `PUT /sensors/:id`,
    - удаление сенсора `DELETE /sensors/:id`,
    - обновление значения сенсора `PATCH /sensors/:id/value`.
- Данные о датчиках находятся в базе данных PostgreSQL.
- Сенсор хранит метаданные: тип, локацию, единицы измерения (unit), статус и временные метки.

**Мониторинг температуры:**

- Пользователь может просматривать текущую температуру в доме через веб-интерфейс:
    - по сенсору (в составе ответа `GET /sensors` и `GET /sensors/:id`),
    - по локации (`GET /sensors/temperature/:location`).
- Для получения текущей температуры используется синхронный HTTP-запрос через внешний Temperature API.
- Хранение истории измерений не реализовано (в базе данных хранится только последнее значение температуры).


### 2. Анализ архитектуры монолитного приложения

### Используемые технологии
- Язык: Go
- База данных: PostgreSQL (доступ через pgxpool)
- Архитектура: Монолитная, все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения.
- Взаимодействие: Синхронное, запросы обрабатываются последовательно.

### Внешние зависимости
- Temperature API — внешний HTTP-сервис, из которого монолит получает актуальные значения температуры.

### Эксплуатационные характеристики
- Масштабируемость ограничена: монолит сложно масштабировать по частям.
- Развёртывание требует остановки всего приложения.

### 3. Определение доменов и границы контекстов

В текущей реализации система представляет собой монолитное приложение без явного разделения на bounded contexts. Однако с точки зрения Domain-Driven Design можно логически выделить доменные области и определить потенциальные границы контекстов.

**Домены текущей системы (AS-IS)**

- Управление датчиками (сенсорами)
- Мониторинг температуры
- Интеграция с внешним Temperature API
- Управление отоплением

**Контекст управления датчиками (Sensor Management)**

Назначение: учёт и администрирование сенсоров в системе.

Сущность:
- Sensor (идентификатор, тип, локация, единицы измерения, статус, временные метки)

Ответственность контекста:
- CRUD-операции над сенсорами
- хранение метаданных сенсора в PostgreSQL
- обновление текущего состояния сенсора

Граница:
- контекст отвечает только за хранение и изменение состояния сенсора
- не содержит сложной бизнес-логики
- не хранит историю измерений

**Контекст мониторинга температуры (Temperature Monitoring)**

Назначение: предоставление пользователю актуальных значений температуры.

Особенности текущей реализации:
- хранится только последнее значение температуры
- не хранит историю измерений
- актуальные данные получаются синхронно через внешний API

Граница:
- отвечает за получение и отображение текущего состояния
- не занимается аналитикой или агрегацией

**Контекст интеграции с внешним API (Temperature API Integration)**

Назначение: взаимодействие с внешним Temperature API.

Реализация:
- сервис TemperatureService
- HTTP-клиент
- преобразование ответа внешнего сервиса во внутреннюю модель

Данный контекст носит инфраструктурный характер и выполняет роль интеграционного слоя.

**Домены целевой системы (TO-BE)**

С учетом бизнес требований и перехода к SaaS-модели можно выделить следующие домены:
- Управление домами и установленными модулями (House & Installed Module Management)
- Каталог модулей (Module Catalog)
- Управление устройствами (Device Management)
- Телеметрия и мониторинг (Telemetry & Monitoring)
- Управление сценариями и автоматизация (Automation & Scenario Management)
- Управление пользователями и доступом (User & Access Management)
- Биллинг и подписки (Billing & Subscription Management)

Таким образом, текущая система является отопление-центричной и ограниченной по функциональности, тогда как целевая экосистема должна стать платформой управления различными модулями умного дома с расширяемой архитектурой.

### **4. Проблемы монолитного решения**

- В обработчике `GET /sensors` выполняются последовательные вызовы внешнего Temperature API для каждого сенсора (риск N+1 вызовов и деградации производительности при росте числа устройств).
- В HTTP-handlers смешаны ответственности (работа с базой данных и интеграция с внешним сервисом выполняются в одном месте).
- Используется `context.Background()` при обращении к БД вместо контекста HTTP-запроса.
- Отсутствует явное разделение слоёв архитектуры (presentation / domain / infrastructure).

## Задание 4. Создание и документирование API

### 1. Тип API

В целевой архитектуре используются два типа взаимодействия между микросервисами:

**REST API** применяется в случаях, когда требуется синхронный запрос-ответ.

Например: получение информации об устройстве, отправка команды устройству, управление пользователями.

**Event-driven взаимодействие** используется для асинхронного обмена событиями между сервисами.

Например: публикация события о новом измерении телеметрии или изменении статуса устройства.

Такой подход позволяет:
- обеспечить предсказуемое поведение пользовательских операций (через REST);
- снизить связанность сервисов и повысить масштабируемость системы (через события).

### 2. Документация API

Для документирования REST API используется Swagger / OpenAPI 3.0

- [Device Management API](./docs/api/device-management.openapi.yaml)

Для описания событийной интеграции используется спецификация AsyncAPI

- [Device Management Service Events](./docs/api/device-management.asyncapi.yaml)

Документация API содержит примеры ключевых эндпоинтов и событий,
демонстрирующих подход к проектированию контрактов взаимодействия.
Полный перечень методов в рамках MVP не реализован и может быть расширен
в дальнейших итерациях разработки.

## Задание 5. Работа с docker и docker-compose

### 1. Приложение temperature-api

Реализовано отдельное приложение temperature-api (Node.js + Express) с эндпоинтом:

`GET /temperature?location=${location}&sensorId=${sensorId}`

Логика:
- если не передан location, он определяется по sensorId;
- если не передан sensorId, он определяется по location;
- при каждом вызове генерируется случайная температура (18–28°C).

Сервис имитирует внешний датчик температуры.

### 2. Контейнеризация приложения

Приложение temperature-api упакован в Docker (порт 8081) и добавлен в docker-compose.yml.

### 3. Настройка базы данных (в docker-compose.yml)

В docker-compose также добавлены:
- сервис postgres (образ postgres:16-alpine);
- инициализация базы через ./smart_home/init.sql;
- переменные окружения для подключения монолита к БД и temperature-api.

## **Задание 6. Разработка MVP**

В рамках задания реализованы два новых микросервиса:

- Device Service — отвечает за управление устройствами (получение списка устройств, создание устройства, изменение статуса).

- Telemetry Service — отвечает за приём и получение телеметрических данных устройств.

Оба сервиса реализованы с простейшей логикой и REST-интерфейсом.
Для хранения данных в MVP используется in-memory хранилище.

### Интеграция с монолитом

Монолитное приложение интегрировано с новым Device Service по REST.

В рамках постепенной декомпозиции обработчик
`GET /api/v1/sensors` был изменён:

- ранее данные получались напрямую из базы данных,
- в MVP данные запрашиваются через HTTP-вызов к device-service.

Таким образом реализован первый шаг по переносу функциональности из монолита во внешний микросервис.

Telemetry Service подготовлен для дальнейшего переноса логики получения и хранения телеметрии.

### Результат

В рамках задания реализован первый этап перехода от монолитной архитектуры к микросервисной модели.

Созданы два новых микросервиса (Device Service и Telemetry Service) и обеспечена их интеграция с существующим монолитным приложением по REST. Это позволило продемонстрировать постепенный перенос функциональности из монолита во внешние сервисы.

Инфраструктура контейнеризации обновлена: все сервисы (монолит, база данных, temperature-api и новые микросервисы) запускаются совместно через docker-compose и взаимодействуют в единой сети.

В рамках MVP данные устройств в Device Service хранятся во временном in-memory хранилище. Полный перенос модели данных и выделение отдельной базы данных для микросервиса предполагаются на следующем этапе миграции архитектуры.
