# Architecture Foundations & Assumptions

Данный документ содержит архитектурные решения, гипотезы и допущения, принятые при проектировании целевой системы.

---

## 1. Центр доменной модели

В качестве центральной сущности системы выбран **House (Дом)**.

Обоснование:
- Пользователь управляет домом
- Устройства принадлежат дому
- Модули устанавливаются в дом
- Автоматизация работает на уровне дома
- Подписки могут применяться к дому или к установленным модулям

---

## 2. Основные домены целевой системы

Выделены следующие доменные области:

- Module Catalog
- Installed Module Management
- Device Management
- Telemetry & Monitoring
- Automation & Scenario Management
- User & Access Management
- Billing & Subscription Management

---

## 3. Принципы декомпозиции

- Каждый домен реализуется отдельным микросервисом
- Каждый микросервис владеет своей базой данных
- Межсервисное взаимодействие преимущественно асинхронное
- Синхронные вызовы используются только при необходимости

---

## 4. Архитектурные принципы

- Event-driven взаимодействие через брокер сообщений
- Слабая связность сервисов
- Изоляция ответственности
- Расширяемость по новым типам устройств
- Поддержка неизвестных будущих модулей

---

## 5. Гипотезы и допущения

- Один дом может иметь несколько установленных модулей
- Один модуль может содержать несколько устройств
- Устройство принадлежит одному дому
- Устройство генерирует множество записей телеметрии
- Пользователь может иметь доступ к нескольким домам
- Подписка может применяться к дому или установленному модулю

---

## 6. Модель взаимодействия

Предполагаемая схема:

- Пользователь взаимодействует через API Gateway
- Микросервисы общаются через message broker
- Телеметрия публикуется событиями
- Automation реагирует на события телеметрии
- Billing реагирует на события активации модулей

---

## 7. Эволюционный переход

- Монолит постепенно интегрируется с новыми сервисами
- Новая функциональность реализуется в микросервисах
- Старые функции переносятся поэтапно
